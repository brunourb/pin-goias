Padrão de Interoperabilidade entre Sistemas

Diretrizes para implementação de serviços para recuperação de informações estratégicas

Objetivo

O presente documento tem como objetivo fornecer as definições básicas para a interoperabilidade entre os sistemas.

Informações básicas do serviço

Informações necessárias para a utilização de um serviço pela SEGPLAN.

  • Nome: Identificador do serviço.
  • Descrição: Descrição e características do serviço.
  • Versão: Versão atual do serviço.
  • Responsável: Informações do responsável pelo serviço no fornecedor.
    1. Órgão: Órgão responsável pelo serviço.
    2. Setor/Área: Área responsável pela manutenção do serviço.
    3. Nome: Nome do responsável pelo serviço (pessoa ou área).
    4. Função: Função do responsável pelo serviço.
    5. Email: Email de contato do responsável pelo serviço.
    6. Telefone: Telefone de contato do responsável pelo serviço (informar todos os disponíveis).

Versionamento de Serviços

O versionamento dos serviços serão especificadas na URL do serviço disponibilizado. As URLs deverão seguir o seguinte padrão:

<Url do Serviço>/<Versão>/<Operações>

onde:

    • • URL do Serviço: - Define a URL base para acesso ao serviço. Por exemplo: http://localhost:8080/pin-api/api .
    • • Versão: - Define a versão do serviço disponibilizada. Este valor deve conter apenas números e os caracteres “.”, “-” ou “_”. Por exemplo: 1.0.0, 1-0-0 ou 1_0_0.
    • • Operações: - As operações do serviço disponibilizado. Por exemplo, “aluno/lista”.

Exemplo:

http://localhost:8080/pin-api/api/1.0.0/aluno/listar

Interface de interoperabilidade

Os serviços deverão ser expostos através de webservices que utilizam o padrão RESTful (Representational State Transfer).

Linguagem de intercâmbio de dados

As informações devem ser disponibilizadas em um dos seguintes padrões de intercâmbio de dados:

JSON(Javascript Object Notation)

Padrão de documentação

Utilização da especificação Open API, no formato JSON

  • http://openapis.org
  • A Open API fornece uma especificação aberta que define metadados para descrever APIs REST, a primeira versão desta especificação foi baseada na especificação 2.0 do Framework Swagger (http://swagger.io/specification)

A especificação OpenAPI foi criada por um consórcio, a Open API Iniciative (OAI), formado por especialistas que reconhecem a importância da padronização de como as APIs REST são descritas. Várias grandes empresas são membros deste consórcio, entre elas, a IBM, Google, Microsoft, SmartBear (que doou a especificação do Swagger 2.0 como base para a primeira versão da especificação OpenAPI), PayPal, entre outros.


Várias dessas empresas já utilizam este padrão como base para a documentação de serviços REST, por exemplo, a IBM, que disponibiliza as APIs do IBM Watson Developer Cloud utilizando o framework Swagger (base da especificação OpenAPI) para a documentação destas APIs disponibilizadas (https://watson-api-explorer.mybluemix.net)

Máscara de atributos

Os serviços disponibilizados podem ser consumidos por aplicações que fornecem uma interface com o usuário. Desta forma, pode ser necessário informações adicionais, como a máscara de apresentação do atributo. Para este caso, esta informação será especificada utilizando uma extensão da especificação definida como “x-mask”, definida na documentação da operação que fornece esta informação.

Recursos e Representações

Seguindo a definição de REST da Universidade da California em Irvine. O recurso é a abstração chave no estilo arquitetural para Apis Web REST. Sendo assim algumas recomendações são feitas para nomeá-los.https://www.ics.uci.edu/~fielding/pubs/dissertation/rest_arch_style.htm .

Deve-se evitar a utilização de verbos para nomear operações em recursos, limitando a utilização de substantivos. Ex: /aluno/inserir.

Essa informação pode ser concluída através do método HTTP POST no recurso Aluno.

Paginação

Caso seja necessário definir uma forma de paginação para o retorno dos dados solicitados ao serviço, deve-se informar as seguintes informações:

  • Dados de Consulta
    1. Quantidade de registros
      1. Nome do Parâmetro: limit
      2. Tipo: Query String
      3. Valor: Inteiro
      4. Valor padrão: definido pelo fornecedor
      5. Exemplo: pessoa?idade=18&limit=1000
    2. Inicio da consulta
      1. Nome do parâmetro: offset
      2. Tipo: Query String
      3. Valor: Inteiro
      4. Valor padrão: 0
      5. Exemplo: pessoa?idade=18&offset=2000&limit=1000
  • Dados de Resposta
    1. Códigos de Status HTTP
      1. Consulta sem retorno de dados: HTTP 204
      2. Consulta com somente uma página: HTTP 200
      3. Consulta com paginação: HTTP 206
      4. Consulta com tamanho dos parâmetros limit e offset acima dos valores padrão: HTTP 416
    2. Cabeçalhos de Resposta:
      1. O cabeçalho de resposta segue o padrão de Content Range, definido pela IETF (https://tools.ietf.org/html/rfc7233)
      2. Paginação atual
        1. Nome do parâmetro: Content-Range
        2. Exemplo: 0-999/10000

Códigos de Retorno de Chamada (HTTP)

Definir os principais códigos de resposta HTTP (https://tools.ietf.org/html/rfc7231#section-6

  • 1xx (Informacional): A requisição foi recebida, continuando o processo
  • 2xx (Sucesso): A requisição foi recebida com sucesso, entendida e aceita
  • 3xx (Redirecionamento): É necessário alguma ação adicional para que a ação seja concluída
  • 4xx (Erro Cliente): A requisição contém sintaxe inválida ou não pode ser realizada
  • 5xx (Erro Servidor): O servidor falhou ao tratar uma requisição aparentemente válida

Códigos de resposta HTTP esperados/retornados:

Código Descrição
HTTP 200 Requisição Completa
HTTP 204 Sem retorno de dados
HTTP 206 Existem mais dados a serem retornados
HTTP 400 Requisição inválida
HTTP 401 Não autorizado
HTTP 403 Sem permissão
HTTP 408 Timeout da requisição
HTTP 416 Intervalo solicitado inválido
HTTP 422 Não foi possível processar o conteúdo da mensagem
HTTP 500 Erro interno no servidor
HTTP 503 Serviço indisponível
HTTP 504 Timeout do Gateway
HTTP 522 Timeout do Servidor

Infraestrutura e Segurança

É necessário que os serviços estejam disponíveis, no mínimo, nos seguintes ambientes:

  • Homologação
  • Produção

Definição dos requisitos de segurança que o serviço deve atender para que seja utilizado:

  • A rede deve estar integrada a rede corporativa do Estado de Goiás.
  • As portas 80 (HTTP) e 443 (HTTPS) devem estar liberadas para comunicação nos servidores supracitados.

Autenticação

A autenticação para a utilização dos serviços deve seguir a especificação OAUTH 2 (http://oauth.net/2/), utilizando o grant_type=client_credentials, para permitir o acesso ao token de autenticação.

Exemplo:

POST https://api.oauth2server.com/token?grant_type=client_credentials&client_id=CLIENT_ID&client_secret=CLIENT_SECRET

Disponibilização do Serviço

Para a disponibilização de novos serviços, devem ser informadas as seguintes informações:

  • Nome: Identificador do serviço;
  • Descrição: Descrição e características do serviço;
  • Versão: Versão atual do serviço;
  • Data para disponibilização do serviço: Data prevista para a disponibilização do serviço;
  • Responsável: Informações do responsável pelo serviço no fornecedor;
    1. Órgão: Órgão responsável pelo serviço;
    2. Setor/Área: Área responsável pela manutenção do serviço;
    3. Nome: Nome do responsável pelo serviço (pessoa ou área);
    4. Função: Função do responsável pelo serviço;
    5. Email: Email de contato do responsável pelo serviço;
    6. Telefone: Telefone de contato do responsável pelo serviço (informar todos os disponíveis).

Para a desativação de serviços, devem ser informadas as seguintes informações:

  • Nome: Identificador do serviço;
  • Versão: Versão atual do serviço;
  • Data Prevista para Desativação do Serviço: Data prevista para que o serviço será desativado;
  • Responsável: Informações do responsável pelo serviço no fornecedor;
    1. Órgão: Órgão responsável pelo serviço;
    2. Setor/Área: Área responsável pela manutenção do serviço;
    3. Nome: Nome do responsável pelo serviço (pessoa ou área);
    4. Função: Função do responsável pelo serviço;
    5. Email: Email de contato do responsável pelo serviço;
    6. Telefone: Telefone de contato do responsável pelo serviço (informar todos os disponíveis).

*As informações do responsável são opcionais e devem ser informadas quando houver alguma atualização da informação.

Atualização de Serviço

Para a disponibilização de novas versões de serviços, devem ser informadas as seguintes informações:

  • Nome: Identificador do serviço;
  • Versão: Versão atual do serviço;
  • Nova Versão: Nova versão a ser disponibilizada;
  • Data de Disponibilização da Versão: Data em que o novo serviço será implantado;
  • Data Prevista para Desativação da Versão Atual: Data em que a versão atual será desativada;
  • Notas da Versão Descrição resumida das alterações da nova versão.
  • Responsável: Informações do responsável pelo serviço no fornecedor;
    1. Órgão: Órgão responsável pelo serviço;
